# Redeploy checklist — audit #2 fixes (2026-06-29)

Two contract fixes need to go on-chain:
- **CRITICAL** — `ArcadeV3SwapRouter` callback drain (`_authorisedPool` guard).
- **HIGH** — `ArcadeLaunchpad` H-1 migration brick (try/catch in `_migrate`).

The router is stateless → a clean surgical redeploy (Path A). The launchpad is
stateful and `setV3Infra` is one-time, so wiring it to a new router or shipping
its H-1 fix means a full stack redeploy (Path B).

---

## Path A — redeploy the V3 router ONLY  (URGENT, do first, ~5 min)

Fixes the CRITICAL fund-drain. Low risk: the router holds no state.

```bash
cd contracts

# 1. Build the 0.7.6 V3 layer so out-v3 has the FIXED router bytecode
FOUNDRY_PROFILE=v3 forge build

# 2. !! unset the v3 profile before the script. The `v3` profile has skip=[]
#    and would try to compile the broken lib/v4-periphery; the DEFAULT profile
#    already skips it. The script is 0.8 and must run under the default profile.
unset FOUNDRY_PROFILE

# 3. Deploy the new router (addresses from web/public/deployments.json)
PRIVATE_KEY=0x<deployer> \
V3_FACTORY_ADDRESS=<addresses.v3Factory> \
ARC_USDC_ADDRESS=<addresses.usdc> \
LAUNCHPAD_ADDRESS=<addresses.launchpad> \
forge script script/RedeployV3Router.s.sol --rpc-url arc_testnet --broadcast --skip 'lib/v4-periphery/**'
# -> logs "NEW ArcadeV3SwapRouter: 0x...."
```

Then:
1. Set **`NEXT_PUBLIC_V3_ROUTER_ADDRESS`** = the new address (Vercel env) and
   update `web/public/deployments.json` (`addresses.v3Router`).
2. Redeploy the frontend (push, or Vercel redeploy).
3. New trades auto-approve + use the FIXED router (`useApproveIfNeeded`).

⚠️ **The OLD router stays drainable for allowances users already granted it.**
On testnet that's acceptable. To be clean, tell users to **revoke** their old
approval (revoke.cash on the old router) — or, since it's testnet, ignore. The
launchpad's internal `v3Router` (creatorBuy) still points at the old router but
is NOT the drain vector (it pays from the launchpad's own funds, never a
victim's standing approval), so it can stay.

## Path B — full stack redeploy  (when convenient / before any real test push)

Ships BOTH fixes (new router + launchpad H-1) in a clean stack. **This abandons
all existing test tokens/curves** (they live on the old launchpad). Acceptable
on testnet.

```bash
cd contracts
forge build                      # 0.8 layer (launchpad etc.)
FOUNDRY_PROFILE=v3 forge build   # 0.7.6 V3 layer

PRIVATE_KEY=0x<deployer> \
ARC_USDC_ADDRESS=<addresses.usdc> \
TREASURY_ADDRESS=<treasury> \
forge script script/DeployTestnet.s.sol --rpc-url arc_testnet --broadcast
```

Then update **every** address in `web/public/deployments.json` + the matching
`NEXT_PUBLIC_*` Vercel envs from the script output, redeploy the frontend, and
re-seed test liquidity. Users re-approve across the board.

---

## Recommendation

Do **Path A now** (kills the CRITICAL drain with minimal disruption). Schedule
**Path B** before the next serious testing round / mainnet so the launchpad H-1
fix and a clean new-router wiring land together. The remaining audit-#2 items
(degenerate-pool grief → pre-create the pair at createToken; per-IP rate limits;
twitter claim deadline vs timelock) are mainnet-hardening, fold them into Path B.
