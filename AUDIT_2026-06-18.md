### C-01. CCTP Iris attestation URL hardcoded to testnet sandbox with no production switch
- **File**: `web/lib/cctp.ts:23, 287, 330`
- **Impact**: On mainnet launch, **every CCTP bridge stalls at "Waiting for Circle attestation"** because `iris-api-sandbox.circle.com` never returns a `complete` payload for a real mainnet burn. M2 of the grant becomes a UI-only feature on day 1.

## HIGH (20)

### Stats integrity (7)

**H-01. WUSDC-paired pools entirely missed by V3 and V2 volume scanners**  
`web/lib/stats.ts:578, 723`. Both `sumV3SwapVolume` and `sumV2SwapVolume` only match `t0/t1 === ADDRESSES.usdc` (native 6-dec 0x3600...). Synthra and UnitFlow route exclusively through WUSDC (0x911b..., 18-dec). Every WUSDC-paired swap is invisible to the public dashboard. Worse: a naive fix that adds WUSDC to the equality check still silently drops everything because the 18-dec amount blows past `MAX_SANE_EVENT_MICROS`. Fix: track both, AND scale WUSDC amounts by `/ 10n**12n` to reach 6-dec USDC parity.

**H-02. MAX_SANE_EVENT_MICROS = 10M USDC ceiling silently drops legitimate large trades**  
`web/lib/stats.ts:473, 530, 689, 834`. Strict `<` against 10M. On mainnet any single $10M+ swap (institutional flow, treasury rebalance) drops with no log. Fix: replace the dollar ceiling with a decoder-correctness check (validate topic[0] matches the ABI, validate data length), and log any drops.

**H-03. uniqueWallets uses topics[1] which is a token/router address on most events**  
`web/lib/stats.ts:243-246`. The "Unique wallets" headline is structurally wrong. Launchpad `Buy/Sell` put TOKEN at topics[1], not the trader. V2/V3 Swap put ROUTER at topics[1] (one address for every trade routed through it). V4 hook `CurveBuy/CurveSell` put bytes32 PoolId at topics[1]. Fix: per-event-signature decode, grab the actual user field (topics[2] for Buy/Sell, recipient for V3 Swap, `to` for V2 Swap). Until shipped, relabel the headline as "unique addresses observed".

**H-04. V4 ArcadeHook CurveBuy/CurveSell volume entirely uncounted**  
`web/lib/stats.ts:260-288`. `sumLaunchpadVolume` filters on V2 launchpad topics. V4 hook emits different signatures (`CurveBuy(bytes32,address,uint256,uint256)`). Every V4 curve trade is invisible. Fix: add `sumHookVolume` keyed on the new topics.

**H-05. MAX across snapshots locks public counters to any single corrupted row forever**  
`web/lib/statsPersistence.ts:98-129`. Only `volume_usdc_micros` is filtered (< 1e15). All 6 other monotonic columns (tx_count, unique_wallets, tokens_launched, v4_tokens_launched, v4_hook_launches, estimated_usdc_gas_micros) are unfiltered. The team already hit this once ($26 quadrillion volume row, see the inline comment) but only patched one column. Fix: bound MAX to last N hours of cron-tagged rows, or add an insert-time sanity gate.

**H-06. AutoCompounder, Orbs limit orders, V2Zap, LockedVault, V4 router/quoter not in tx/wallet scan**  
`web/lib/stats.ts:184-201`. The `contracts` array is missing every recently shipped flagship surface. Any tx that only touches AutoCompounder, Orbs TWAP, V2Zap, LockedVault, or the V4 prototype router is invisible. Fix: add optional-gated entries matching the existing twitterEscrow/v4Launchpad pattern.

**H-07. /stats methodology text claims "Arcade V3 pool Swap" but scanner is address-less and counts every fork**  
`web/app/stats/page.tsx:181-188`. The methodology copy describes a scope that does not match the scanner's actual behavior (it's address-less, captures every V3 pool on Arc including Rubicon, Synthra, UnitFlow). Fix: rewrite the methodology paragraph to be honest about the address-less scan semantics.

### CCTP (1)

**H-08. Attestation polling has zero backoff and ignores the detailed transient/missing signal**  
`web/components/bridge/BridgeCard.tsx, web/components/bridge/BridgeHistory.tsx`. Iris polled at fixed cadence with no exponential backoff. On mainnet Iris egress slots get hammered, and a transient "missing" response is treated identically to a hard error. Fix: backoff with jitter, treat `status: "missing"` and `status: "transient"` as distinct from `status: "failed"`.

### ERC-8004 Identity (4)

**H-09. ERC-8004 `mint()` has no on-chain tier enforcement — anyone can self-issue Diamond**  
`web/lib/abis/erc8004Identity.ts`. The registry contract has no tier concept. The Silver/Gold/Diamond gate exists ONLY in the frontend `useCreatorTier.ts` hook. A user can call `mint()` directly with a hand-crafted Diamond metadata URI. Result: anyone gets a Diamond NFT, the cross-app reputation primitive is meaningless. Fix: deploy an `ArcadeIdentityIssuer` wrapper that verifies tier on-chain against the launchpad before forwarding to the Registry, OR pivot the UX to make this an explicit anyone-can-mint identity NFT and remove the "earned tier" framing.

**H-10. Tier upgrade leaves stale Silver/Gold metadata permanently baked into the data: URI**  
`web/components/creator/DiamondIdentityMint.tsx`. Metadata is hex-encoded data URI inline at mint time. Creator hits Silver, mints. Later graduates to Gold/Diamond. The old NFT still shows Silver because data URIs are immutable. Fix: either (a) use a `tokenURI` route that reads tier server-side at view time, OR (b) burn-and-re-mint on upgrade, OR (c) document the static-tier semantic explicitly so users don't expect upgrades.

**H-11. V4 hook (ArcadeHook) launches are not counted toward any tier**  
`web/lib/hooks/useCreatorTier.ts`. The tier counter iterates V2 launchpad generations only. V4 hook launches via `arcadeHook.createLaunch` never increment the count. Creators who exclusively use V4 stay at "none" forever. Fix: add a V4-hook events query (`TokenLaunched` topic) to the count.

**H-12. CLANKER_V3 launches count as "bonded" at creation — Diamond mintable for 30 USDC**  
`web/lib/hooks/useCreatorTier.ts`. The logic `(mode === 2 || migrated)` treats any CLANKER_V3 launch as bonded from block 1 because CLANKER_V3 has no curve to bond. Creation fee is 3 USDC. To hit Diamond (10 launches), pay 30 USDC and you get a Diamond Creator Identity NFT. Same pattern for Silver (3 launches, 9 USDC) and Gold (5 launches, 15 USDC). Fix: either require CLANKER_V3 launches to have non-zero LP liquidity OR raise the threshold OR don't count CLANKER_V3 toward tier counts.

### API security (2)

**H-13. Unauthenticated debug route walks every V2 pair (RPC quota burn + private-data exposure)**  
`web/app/api/debug/v2-lp-positions/route.ts`. No auth, no rate limit, returns every LP position. Anyone can DoS the Arc RPC by hitting this in a loop. Fix: gate behind `STATS_CRON_SECRET` (or similar bearer) AND `rateLimit()`. Or move it under `/api/debug/` with NODE_ENV=development guard.

**H-14. Auto-claim cron reads `pendingClaims` tuple at wrong indices, breaks the entire auto-claim path**  
`web/app/api/twitter/cron/route.ts`. The contract returns a struct; the route destructures it at the wrong indices, so the cron has been silently no-op-ing every run. F2 auto-claim is currently dead. Fix: re-derive the indices from the ABI and add a smoke test.

### EIP-712 Twitter Escrow (1)

**H-15. Compromised backend key drains escrow in a single block: testnet `MIN_TIMELOCK` / `DEFAULT_TIMELOCK` are 0 and admin UI cannot rotate the signer**  
`contracts/src/launchpad/ArcadeTwitterEscrowV3.sol`. On testnet the claim timelock defaults to 0, which means a compromised backend key can sign and immediately execute claims with no delay. Compounding: the admin UI's `Rescue` card surfaces `setTrustedSigner` (deprecated, reverts USE_TIMELOCK_ROTATION), so an operator who notices a compromise cannot rotate the signer through the UI. Fix: (a) bump `MIN_TIMELOCK` to 1h even on testnet, (b) replace the admin UI's `setTrustedSigner` call with the 2-step `requestTrustedSignerRotation` + `finalizeTrustedSignerRotation` flow, (c) surface the cancel path.

### Swap card / routing (4)

**H-16. V3 exact-output path silently broken: typed `For` amount is back-derived from a 1-unit probe and submitted as exact-IN with weak slippage protection**  
`web/components/swap/SwapCard.tsx`. The "I want exactly X out" path uses a 1-unit probe to derive a swap rate, multiplies by the desired output, and submits as `exactInput` with that derived amount. For thin pools the 1-unit probe rate is meaningless, and the actual output can be 10-50% off from what the user typed. Fix: implement a proper binary search over `quoteExactInputSingle` until target output is matched within tolerance, OR cap the probe's confidence and disable the exact-output path on pools where the 1-unit quote is below a minimum threshold.

**H-17. `selectedRoute` persists across `amountIn` changes — submit uses stale `executor.args` with the old amount**  
`web/components/swap/SwapCard.tsx`. Changing the input amount does not invalidate the previously-selected route's executor args. User types 10 USDC, picks route A, changes to 1000 USDC, clicks swap — the tx still uses the executor args computed for 10 USDC. Fix: reset `selectedRoute` to undefined on `amountIn` change OR re-compute executor args on submit instead of caching.

**H-18. Impact-aware auto-pick uses cross-provider reference quote as fallback, producing wildly wrong impact numbers and silently routing users away from the actually-best route**  
`web/components/swap/SwapCard.tsx`. The per-route impact percent is computed against `refQuotes.best`, which is the best output across ALL providers. A genuinely good route reports >0% impact against the best (itself), and a worse route reports lower (negative) impact because it's compared against the same reference. The auto-pick then routes the user away from the best route. Fix: compute impact per-route using the route's OWN small-amount baseline, not the cross-provider best.

**H-19. Auto-picked impact-aware route is sticky across amount changes — typing larger amount does not re-evaluate route choice**  
`web/components/swap/SwapCard.tsx`. Same root cause as H-17. Auto-pick fires once when quotes land. User then types a larger amount, all routes recompute, but the auto-picked selection doesn't re-evaluate, so the user stays on a now-suboptimal route silently. Fix: re-run impact-aware selection whenever `routeQuotes.quotes` array reference changes.

### Frontend (1)

**H-20. Historical bonding-curve candle price is missing the 10^12 USDC/token scale that the live WS path applies — every chart jumps by 12 orders of magnitude when a live trade arrives**  
`web/lib/hooks/useTokenCandles.ts`. Historical candles compute price in raw `usdc_micros / token_wei` units. Live WS candles apply the 10^12 scale to normalize. When the live socket pushes the first tick, the chart jumps 12 orders of magnitude. Fix: apply the same `* 10n**12n` scaling in the historical decoder.
